═══════════════════════════════════════════════════════════════
        RAG PROJECT — TECH STACK LEARNING GUIDE
═══════════════════════════════════════════════════════════════

This guide covers every technology used in the RAG project
and what you should learn about each one.

Suggested Learning Order:
  1. Python basics
  2. FastAPI
  3. PostgreSQL + SQLAlchemy
  4. Docker
  5. Redis
  6. Celery
  7. LLMs + Embeddings
  8. Qdrant
  9. CI/CD (GitHub Actions)

───────────────────────────────────────────────────────────────
1. FASTAPI
───────────────────────────────────────────────────────────────
What it is:
  Modern Python web framework for building APIs.
  Used in this project as the main API layer.

Must know:
  - What is an API and HTTP methods (GET, POST, PUT, DELETE)
  - Path parameters vs query parameters vs request body
  - Pydantic models for request/response validation
  - Dependency injection (Depends())
  - Middleware and CORS
  - lifespan context manager (startup/shutdown events)
  - Async vs sync endpoints (async def vs def)
  - Background tasks

Resources:
  - https://fastapi.tiangolo.com/tutorial/


───────────────────────────────────────────────────────────────
2. POSTGRESQL
───────────────────────────────────────────────────────────────
What it is:
  Relational database. Stores users, documents, and metadata.
  Hosted on Supabase in production.

Must know:
  - Tables, rows, columns, primary keys, foreign keys
  - Basic SQL (SELECT, INSERT, UPDATE, DELETE)
  - Joins (INNER, LEFT)
  - Indexes and why they matter for performance
  - Transactions (ACID properties)
  - SQLAlchemy ORM — models, sessions, queries
  - Alembic migrations — what they are and why needed
  - asyncpg — async driver for non-blocking DB calls

Resources:
  - https://www.postgresql.org/docs/
  - https://docs.sqlalchemy.org/


───────────────────────────────────────────────────────────────
3. REDIS
───────────────────────────────────────────────────────────────
What it is:
  In-memory key-value store.
  Used for caching (exact match + semantic) and Celery broker.
  Hosted on Upstash in production.

Must know:
  - What is caching and why it matters
  - Key-value concept (SET key value, GET key)
  - TTL (Time To Live) — auto-expiring keys
  - Redis as a message broker (pub/sub)
  - Difference between cache hit and cache miss
  - When to use Redis vs PostgreSQL

Resources:
  - https://redis.io/docs/


───────────────────────────────────────────────────────────────
4. QDRANT
───────────────────────────────────────────────────────────────
What it is:
  Vector database. Stores embeddings for semantic search.
  Hosted on Qdrant Cloud in production.

Must know:
  - What are vectors and embeddings
  - How semantic search works vs keyword search
  - Cosine similarity — how similarity is measured
  - Collections, points, payloads in Qdrant
  - score_threshold — filtering low quality results
  - Difference between vector DB and relational DB
  - Why RAG needs a vector DB

Resources:
  - https://qdrant.tech/documentation/


───────────────────────────────────────────────────────────────
5. MINIO / SUPABASE STORAGE
───────────────────────────────────────────────────────────────
What it is:
  Object storage. Stores PDF files (like AWS S3).
  MinIO used locally, Supabase Storage used in production.

Must know:
  - What is object storage vs file system vs database storage
  - Buckets and objects concept
  - S3-compatible API (PUT, GET, DELETE)
  - Presigned URLs — temporary access links
  - When to use object storage vs database for files

Resources:
  - https://min.io/docs/
  - https://supabase.com/docs/guides/storage

6. CELERY
───────────────────────────────────────────────────────────────
What it is:
  Background task queue. Processes PDFs asynchronously
  so the API doesn't block while embedding documents.

Must know:
  - What is a task queue and why needed
  - Difference between sync and async processing
  - Tasks, workers, brokers, backends
  - .delay() vs .apply_async()
  - Task states (PENDING, STARTED, SUCCESS, FAILURE)
  - --pool=solo on Windows (multiprocessing limitation)
  - Flower — monitoring Celery tasks via web UI

Resources:
  - https://docs.celeryq.dev/

───────────────────────────────────────────────────────────────
14. RAG PIPELINE (CORE CONCEPT)
───────────────────────────────────────────────────────────────
What it is:
  Retrieval-Augmented Generation. The core architecture
  of this entire project.

Must know:
  - What is RAG and why it's better than pure LLM
  - Chunking — why we split documents into pieces
  - Chunk size and overlap — how they affect accuracy
  - Embedding — converting text to vectors
  - Retrieval — finding relevant chunks via similarity
  - Augmentation — adding context to the LLM prompt
  - Generation — LLM produces answer from context
  - Exact cache vs semantic cache — performance optimization
  - Hallucination — what it is and how RAG reduces it

Full RAG flow in this project:
  Upload PDF
    -> Store in Supabase Storage
    -> Celery task: chunk text (1500 chars, 200 overlap)
    -> Embed chunks with nomic-embed-text (Ollama)
    -> Store embeddings in Qdrant
    -> User asks question
    -> Check exact cache (SHA256) -> return if hit
    -> Check semantic cache (cosine >= 0.95) -> return if hit
    -> Embed question
    -> Retrieve top chunks from Qdrant (score >= 0.55)
    -> Send chunks + question to Groq LLM
    -> Return answer + store in cache

Resources:
  - https://arxiv.org/abs/2005.11401 (original RAG paper)
